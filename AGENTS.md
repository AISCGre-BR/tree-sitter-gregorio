## 1. Project Overview

`tree-sitter-gregorio` is a [tree-sitter](https://tree-sitter.github.io/tree-sitter/)
grammar for the **GABC** and **NABC** notation languages used by
[Gregorio](https://gregorio-project.github.io/), a software suite for typesetting
Gregorian chant in LaTeX.

The grammar parses `.gabc` files into a concrete syntax tree (CST). It is consumed by:

- **`gregorio-lsp`** (sibling repo at `../gregorio-lsp`) — a language server that uses
  the tree optionally via the `tree-sitter` Cargo feature.
- Editor plugins (Helix, Neovim, etc.) using the standard tree-sitter query interface.

**Language specs** live in `docs/`:

| File | Contents |
|---|---|
| `docs/GABC_SYNTAX_SPECIFICATION.md` | Complete GABC notation reference |
| `docs/NABC_SYNTAX_SPECIFICATION.md` | Complete NABC (adiastematic) notation reference |
| `docs/GREGORIO_COMPILER_ERRORS_AND_WARNINGS.md` | Upstream compiler diagnostics |
| `docs/ERRORS_AND_WARNINGS_SUMMARY.md` | Summary table of errors/warnings |
| `docs/HEADER_HIGHLIGHTING.md` | Header highlighting notes |
| `docs/HELIX_SETUP.md` | Helix editor integration guide |
| `docs/LATEX_INJECTION.md` | LaTeX injection query notes |
| `docs/PROJECT_STRUCTURE.md` | Detailed file layout |

## 3. Development Workflow

> **NixOS environment**: `cargo`, `node`, and `tree-sitter` CLI are not in PATH directly.
> Always use nix shells:

```bash
# Generate parser after editing grammar.js
nix shell nixpkgs#nodejs nixpkgs#tree-sitter -c tree-sitter generate

# Run corpus tests
nix shell nixpkgs#nodejs nixpkgs#tree-sitter -c tree-sitter test

# Run a single corpus test file
nix shell nixpkgs#nodejs nixpkgs#tree-sitter -c tree-sitter test -f "pattern"

# Build and test the Rust crate (standalone)
nix shell nixpkgs#cargo nixpkgs#rustc nixpkgs#gcc -c \
  cargo test --manifest-path /path/to/tree-sitter-gregorio/Cargo.toml

# Build and test gregorio-lsp with tree-sitter feature
nix shell nixpkgs#cargo nixpkgs#rustc nixpkgs#gcc -c \
  cargo test --manifest-path /path/to/gregorio-lsp/Cargo.toml --features tree-sitter
```

**The cycle for any grammar change:**

1. Edit `grammar.js`
2. `nix shell nixpkgs#nodejs nixpkgs#tree-sitter -c tree-sitter generate`
3. Add corpus test in `test/corpus/` (or update existing)
4. `nix shell nixpkgs#nodejs nixpkgs#tree-sitter -c tree-sitter test`
5. If new node kinds affect highlighting: update `queries/highlights.scm`
6. If new node kinds affect diagnostics: update `queries/diagnostics.scm`
7. If new node kinds affect LaTeX injection: update `queries/injections.scm`
8. If adding/removing/renaming named node kinds: bump `STABLE_NODE_KIND_CONTRACT_VERSION`
   in `bindings/rust/lib.rs`
9. Run `cargo test` via nix to validate Rust bindings

## 4. Grammar DSL Patterns

The grammar uses the [tree-sitter JavaScript DSL](https://tree-sitter.github.io/tree-sitter/creating-parsers).

### 4.1 Naming Conventions

| Pattern | Meaning |
|---|---|
| `snake_case` | Public named rule (appears in the CST) |
| `_snake_case` (leading underscore) | Inline rule (transparent in CST — children are inlined into parent) |
| `gabc_` prefix | Rules specific to GABC square-note notation |
| `nabc_` prefix | Rules specific to NABC adiastematic notation |
| `header_` prefix | Header section rules |
| `syllable_` prefix | Syllable text / style tag rules |
| `_gabc_attribute_*` | Inline rules for individual `[key:value]` attributes |
| `_nabc_*` | Inline rules for NABC sub-structures |

### 4.2 Token Precedence

- `token(prec(N, str))` — use for literal strings that would otherwise be ambiguous
  (e.g., clef names `c1`–`c4` need `prec(20)` to beat single-letter pitch matches).
- `token.immediate(str)` — matches only when there is no whitespace before the token
  (used for note modifiers like `~`, `<`, `>`, `_`, `.`, shape letters, liquescence).
- `prec.left(N, seq(...))` — use for left-recursive fusion chains (e.g., `gabc_neume_fusion`,
  `gabc_clef_link`, `nabc_glyph_fusion`).

### 4.3 `extras`

Whitespace (`/\s/`) and `comment` (`% …`) are in `extras`, meaning they are silently
consumed between any two tokens globally — **except** inside `token.immediate()` patterns.

### 4.4 `conflicts`

The `conflicts` array in `grammar.js` lists ambiguous rule pairs that tree-sitter's
GLR parser resolves at runtime. Current conflicts include:

- `gabc_separation_bar` — bar modifiers can be ambiguous
- `_gabc_line_break` — custos modifiers can be ambiguous
- `gabc_snippet` — repetition ambiguities with symbols
- `gabc_separation_bar` vs `nabc_spacing` — backtick used by both
- `gabc_neume` — extra signs are optional
- `syllable` — `repeat1` inside tags
- All cross-syllable style/control tag pairs (open vs. complete vs. close variants)

When adding rules that introduce new conflicts, add the conflicting rule pair to the
`conflicts` array. Do **not** remove existing conflict entries unless you are certain
the ambiguity is resolved by another means.

### 4.5 `alias`

`alias(token, $.node_kind)` is used extensively to give anonymous literal tokens a
named node kind in the CST. Example:

```js
alias('c1', $.c_clef)   // the literal string 'c1' appears as node kind 'c_clef'
alias('~', $.diminutive_liquescence)
```

This avoids polluting the grammar with dozens of trivial named rules.

### 4.6 Field Names

Use `field('name', rule)` to annotate semantically important children. This makes
queries more robust (e.g., `(gabc_neume pitch: (pitch_lowercase))` in `.scm` files).
Current field names in use: `pitch`, `shape`, `type`, `name`, `value`, `text`,
`position`, `modifier`, `left`, `right`, `dimensions`, `hint`, `factor`,
`shorthand`, `variant`, `basic_glyph_descriptor`, `glyph_modifiers`, `pitch_descriptor`.

## 5. Key Grammar Sections

### 5.1 File Structure

```
source_file
  ├── header_section    (repeat1 of header)
  ├── section_separator ("%%")
  └── notation_section  (repeat1 of notation_item)
```

Each `notation_item` is an optional `syllable` + mandatory `note_group`.
A `note_group` is `(` + optional `snippet_list` + `)`.

### 5.2 Snippet List (GABC/NABC Alternation)

`snippet_list` holds either a single `gabc_snippet` or an alternating sequence
`gabc_snippet | nabc_snippet [ | gabc_snippet | nabc_snippet … ]`.

The private rule `_alternating_snippet` uses `prec(2)` for NABC and `prec(1)` for
GABC to resolve the alternation ambiguity.

Empty snippets (pipes adjacent to each other or at the boundaries, e.g., `(|vi)`,
`(g||ta)`) are supported as of grammar v0.5.0.

### 5.3 GABC Neume

A `gabc_neume` consists of:
1. Optional initio debilis prefix (`-`)
2. Optional shape prefix (quilisma `w`, oriscus `o`/`O`, virga variants, etc.)
3. Pitch letter (lowercase `a–m` or uppercase `A–M` for punctum inclinatum)
4. Optional orientation suffix (`0`/`1`)
5. Optional liquescence suffix (`~`, `<`, `>`)
6. Optional leaning suffix (`0`/`1`/`2`)
7. Optional `r` for cavum (hollow)
8. Zero or more `gabc_extra_symbol` (punctum mora `.`, ictus `'`, episema `_`, accents)
9. Zero or more `gabc_attribute` `[…]` blocks
10. Optional alteration

### 5.4 GABC Attributes

All `[key:value]` forms are implemented as individual `_gabc_attribute_*` inline rules
and collected by `gabc_attribute`. Current attribute keywords:
`shape`, `nocustos`, `cs`, `cn`, `ob`, `ub`, `ocb`, `ocba`, `ll`, `oll`, `ull`,
`oslur`, `uslur`, `oh`, `uh`, `alt`, `nv`, `gv`, `ev` (verbatim TeX).
Macro references use `[nm0–9]`, `[gm0–9]`, `[em0–9]`, `[altm0–9]`.

### 5.5 NABC

NABC snippets contain `nabc_complex_glyph_descriptor` nodes, each comprising:
- Optional `nabc_spacing` (`//`, `/`, ` `` `, `` ` ``)
- A `nabc_glyph_descriptor` or `nabc_glyph_fusion`
- Zero or more `nabc_subpunctis_prepunctis_descriptor` (`su`/`pp` + modifier + count)
- Zero or more `nabc_significant_letter_descriptor` (`ls…N` or `lt…N`)

Basic glyph codes: `vi`, `pu`, `ta`, `gr`, `cl`, `pe`, `po`, `to`, `ci`, `sc`,
`pf`, `sf`, `tr`, `st`, `ds`, `ts`, `tg`, `bv`, `tv`, `pr`, `pi`, `vs`, `or`,
`sa`, `pq`, `ql`, `qi`, `pt`, `ni`, `un`, `oc`.

## 6. Test Corpus Format

Test files live in `test/corpus/` and use the standard tree-sitter corpus format:

```
================================================================================
Test case title (must be unique within the file)
================================================================================

<input GABC here>

--------------------------------------------------------------------------------

(expected_cst_sexp_here)

================================================================================
Another test case
================================================================================
...
```

Rules:
- `===` lines are separators (exactly 80 `=` characters).
- `---` lines separate input from expected output (at least 1 `-`).
- Expected output is an S-expression matching the tree-sitter node printer format.
- To assert an ERROR node exists: include `(ERROR)` in the expected S-expression.
- Run `tree-sitter test` (or `tree-sitter test -f "title pattern"`) to verify.
- Number files sequentially (`26-new-feature.txt`) to keep them ordered.
- Each file should test one logical concept.

## 7. Node Kind Contract

`bindings/rust/lib.rs` exposes:

```rust
pub const STABLE_NODE_KIND_CONTRACT_VERSION: &str = "1";
```

**Bump this version** whenever you:
- Rename a public named rule (any rule without leading `_`)
- Remove a public named rule
- Change the meaning/structure of an existing public rule in a breaking way

Do NOT bump for:
- Adding new rules (additive changes)
- Fixing parser bugs that don't change the surface API
- Updating queries

Downstream consumers (`gregorio-lsp`) check this version at startup to detect
incompatible grammar updates.

## 8. Queries

### `queries/highlights.scm`
Tree-sitter highlight captures. Standard capture names: `@keyword`, `@constant`,
`@number`, `@string`, `@comment`, `@punctuation.special`, `@attribute`, `@operator`,
`@label`, `@variable`. Add new node captures here when adding new named node kinds
that are visually meaningful.

### `queries/injections.scm`
Language injection for LaTeX fragments. Currently injects `latex` into:
- `tex_code` nodes (verbatim TeX in `[nv:]`, `[gv:]`, `[ev:]` attributes)
- `tex_code_header` nodes (TeX-valued headers like `annotation`, `mode-modifier`)

### `queries/diagnostics.scm`
Error/warning captures for `gregorio-lsp`:

```scheme
(ERROR) @error.syntax
(snippet_list first: (gabc_snippet) "|" (ERROR) @warning.alternation)
```

Add new diagnostic captures here for semantic issues the grammar can detect
structurally (e.g., impossible alternation patterns).

## 9. Common Pitfalls

1. **`token.immediate()` is required for zero-space modifiers**: Note modifiers (`~`,
   `<`, `>`, `_`, shape suffixes) must use `token.immediate()` to prevent the `extras`
   whitespace rule from consuming any space between the pitch and the modifier.

2. **Clef precedence**: Clef names (`c1`, `c2`, `f3`, etc.) must use `token(prec(20, ...))` 
   to beat pitch letter matches (`c`, `f`). Without this, `c4` parses as pitch `c`
   followed by pitch `4` (which is invalid but the parser would try anyway).

3. **NABC backtick conflict**: The backtick character `` ` `` is used both as `virgula`
   in GABC separation bars and as `inter_element_space_left` in NABC spacing. This
   conflict is declared in `conflicts` and resolved by GLR.

4. **Cross-syllable styles**: Tags like `<b>` can span multiple syllables. The grammar
   models this with three variants per tag: complete (`syllable_style_bold`),
   open-only (`syllable_style_bold_open`), and close-only (`syllable_style_bold_close`).
   All three must be listed in `conflicts` against each other.

5. **Empty note groups**: `()` is a valid note group (no notes, no snippets). The
   `snippet_list` inside `note_group` is optional.

6. **Empty GABC/NABC snippets**: In multi-voice NABC scores, snippets separated by
   `|` may be empty: `(|vi)`, `(|vi|ta)`, `(g||ta)`. This is valid since Gregorio 6.2.0
   (fix for issue #1700). The grammar models this by making `gabc_snippet` and
   `nabc_snippet` optional in the alternation positions.

7. **Tilde `~` disambiguation**: `~` inside note parentheses `()` is `diminutive_liquescence`
   (a note modifier). `~` in syllable text (outside `()`) is a `lyric_tie` token
   as of Gregorio 6.2.0. Do not conflate the two.

8. **`prec.left` for fusion chains**: Neume fusion (`@`), clef links (`@`), and NABC
   glyph fusion (`!`) are left-recursive. Without `prec.left(1, ...)` the parser would
   infinite-loop. Always add `prec.left` to any left-recursive binary operator rule.

9. **`repeat1` vs `repeat`**: `repeat1` requires at least one item and is used for
   mandatory sequences (e.g., `notation_section`, `nabc_snippet`). Use `repeat` for
   zero-or-more (e.g., `gabc_extra_symbol` after a pitch).

10. **After `tree-sitter generate`, commit `src/`**: The generated files in `src/`
    (`parser.c`, `grammar.json`, `node-types.json`) must be committed alongside
    `grammar.js`. Do not `.gitignore` them.

## 10. Gregorio Version Compatibility

The grammar targets **Gregorio ≥ 6.2.0**. Key syntax additions by version:

| Version | GABC/NABC Changes |
|---|---|
| 4.0.0 | NABC (`|` alternation), neume fusion (`@`), clef links, half-spaces, slurs, verbatim TeX |
| 4.1.0 | Spacing `/0`, `/!`, `/[factor]`; `[nocustos]`; `[ll:]`; musica ficta `r6–r8`; `<eu>`, `<e>`, `<pr>` |
| 4.2.0 | `[oll:]`/`[ull:]` custom ledger; `<c>` color tag; NABC pitches `hn`/`hp` |
| 5.0.0 | `<clear>`, `<pr:factor>` protrusion; hollow oriscus `r`; `[[` / `]]` brackets |
| 5.1.0 | Virgula/minima on upper ledger (`` `0 ``, `,0`); dotted bar (`:?`); slur attributes `[oslur:]`/`[uslur:]` |
| 5.2.0 | Divisio minimis (`^`); punctum inclinatum variant `2` (uppercase + `2`) |
| 6.0.0 | Parenthesized alterations (`x?`, `y?`, `#?`, `` `? ``, `,?`); `$` escape prefix |
| 6.1.0 | Soft flat `X`, sharp `##`, soft natural `Y` alterations; `[shape:stroke]` hint |
| 6.2.0 | Lyric tie `~` in text; `<sp>ut</sp>`, `<sp>ot</sp>`, `<sp>dt</sp>` tie symbols; empty GABC/NABC snippets |

## 11. Companion Project: gregorio-lsp

`tree-sitter-gregorio` is consumed by **gregorio-lsp** (sibling directory
`../gregorio-lsp`), a Language Server Protocol implementation in Rust. When
making grammar changes, also check:

- `../gregorio-lsp/src/parser/tree-sitter-integration.rs` — uses node kinds and
  field names from the grammar for diagnostic extraction and incremental parsing.
- `../gregorio-lsp/Cargo.toml` — depends on this crate via `path = "../tree-sitter-gregorio"`
  (with optional feature `tree-sitter`).
- If `STABLE_NODE_KIND_CONTRACT_VERSION` is bumped, update the version check in
  `gregorio-lsp` accordingly.

After grammar changes, run the full test suite in `gregorio-lsp`:

```bash
nix shell nixpkgs#cargo nixpkgs#rustc nixpkgs#gcc -c \
  cargo test --manifest-path ../gregorio-lsp/Cargo.toml --features tree-sitter
```

- Scope (optional): `feat(grammar):`, `fix(nabc):`, `test(corpus):`, `docs(agents):`

  ```bash
  git -c gpg.program=gpg -c commit.gpgsign=true commit \
    --gpg-sign=BAC0B1B569777A733E37447FB10712C404063D38 -m "feat(grammar): ..."
  ```
- Keep messages in English.
- Reference Gregorio upstream issues when implementing syntax from a specific release:
  `feat(grammar): add lyric tie ~ in syllable text (gregorio#1684)`

- [ ] Read the relevant section of `docs/GABC_SYNTAX_SPECIFICATION.md` or
      `docs/NABC_SYNTAX_SPECIFICATION.md`
- [ ] Add rule(s) to `grammar.js` following naming conventions (§4.1)

      `bindings/rust/lib.rs`

- [ ] Run `cargo test --features tree-sitter` via nix for `gregorio-lsp`
- [ ] Update `CHANGELOG.md` with a concise entry
